# 引数解釈と分類ルール

## 自然言語キーワード

- 「機能追加」「新機能」: `feat`
- 「修正」「バグ修正」: `fix`
- 「リファクタリング」「整理」: `refactor`
- 「ドキュメント」: `docs`
- 「テスト」: `test`
- 「設定」「依存関係」: `chore`
- 「まとめて」: 同タイプを統合（`GROUPING=merge`）
- 「分けて」「ファイルごと」: 分割（`GROUPING=split`）
- 「計画だけ」「実行しない」: Phase 1 の計画提示で終了（`PLAN_ONLY=true`）
- 「自動で」「自動実行」: Phase 2 の各コミット前確認を省略（`AUTO_EXECUTE=true`）

## フラグ

- `--file <path>`: 特定パスのみ対象
- `--type <type>`: 指定タイプのみ対象
- `--auto`: Phase 1 の計画承認後、各コミット前確認を省略

## 実行意図の定義

- `PLAN_ONLY=true`: 計画を提示して終了する。`y` / `e` / `n` の実行確認は出さない。
- `PLAN_ONLY=false`: 計画提示後に `y` / `e` / `n` で確認する。
- `AUTO_EXECUTE=true`: `y` 承認後に限り、各コミット直前の再確認を省略する。

`--auto` や「自動で」があっても、Phase 1 後の `y` / `e` / `n` 確認は省略しない。

## 意図分類の定義

- `feat`: 新機能、新規コンポーネント
- `fix`: バグ修正、エラーハンドリング
- `refactor`: 構造改善、責務分離、リネーム
- `docs`: 仕様書、README、コメント
- `test`: 単体/統合/E2E テスト追加・修正
- `chore`: 設定変更、依存更新、運用タスク

## 分割ルール

1. 1コミット = 1つの意図
2. 同一ファイルに複数意図が混在する場合は、部分適用で分割
3. 実行順はビルド・テストが壊れにくい順序を優先

## 対象範囲の扱い

- `--file <path>` 指定時は、差分確認、計画、stage の対象をその pathspec に限定する。
- 対象外に staged 差分がある場合は、計画に含めず、実行前にユーザーへ確認する。
- untracked file は `git ls-files --others --exclude-standard` で検出し、計画に含めるか確認できない場合は停止する。
- ignore されているファイルは対象外とし、`git add -f` は使わない。

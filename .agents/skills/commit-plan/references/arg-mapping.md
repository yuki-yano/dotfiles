# 引数解釈と分類ルール

## 自然言語キーワード

- 「機能追加」「新機能」: `feat`
- 「修正」「バグ修正」: `fix`
- 「リファクタリング」「整理」: `refactor`
- 「ドキュメント」: `docs`
- 「テスト」: `test`
- 「設定」「依存関係」: `chore`
- 「まとめて」: 同タイプを統合（`GROUPING=merge`）
- 「分けて」「ファイルごと」: 分割（`GROUPING=split`）
- 「自動で」「自動実行」: 承認後の再確認を省略（`AUTO_EXECUTE=true`）

## フラグ

- `--file <path>`: 特定パスのみ対象
- `--type <type>`: 指定タイプのみ対象
- `--auto`: 承認後は各コミット前確認を省略

## 意図分類の定義

- `feat`: 新機能、新規コンポーネント
- `fix`: バグ修正、エラーハンドリング
- `refactor`: 構造改善、責務分離、リネーム
- `docs`: 仕様書、README、コメント
- `test`: 単体/統合/E2E テスト追加・修正
- `chore`: 設定変更、依存更新、運用タスク

## 分割ルール

1. 1コミット = 1つの意図
2. 同一ファイルに複数意図が混在する場合は、部分適用で分割
3. 実行順はビルド・テストが壊れにくい順序を優先

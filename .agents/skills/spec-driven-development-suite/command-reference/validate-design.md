# validate-design

## 目的

設計品質をレビューし、実装移行の可否を判定する。

## 入力

- 必須: `feature_name`

## 前提確認

1. `./docs/sdd/specs/<feature-name>/design.md` が存在すること。
2. `./docs/sdd/specs/<feature-name>/requirements.md` と `spec.json` が参照可能であること。
3. `./docs/sdd/steering/` のコア/カスタム文書が参照可能であること。

## レビュー方法論

1. フォーカス: 重大課題のみ（最大3件）
2. 形式: 一方向評価ではなく対話型レビュー
3. 結果: GO/NO-GO を明示判定

## コア評価観点

### 1. 既存アーキテクチャ整合（最重要）

- 境界/レイヤ整合
- 既存パターンとの整合
- 依存方向と結合度の妥当性
- モジュール責務との整合

### 2. 設計一貫性と標準

- 命名規約・実装規約への準拠
- エラー処理・ログ戦略の一貫性
- 設定/依存管理の統一性
- データモデル規約との整合

### 3. 拡張性と保守性

- 将来変更への柔軟性
- 関心分離と単一責務
- テスト容易性とデバッグ容易性
- 文書可読性

### 4. 型安全とインターフェース設計

- 型定義の適切性
- `any` 等の危険パターン回避
- API境界の明確性
- 入力検証とエラーハンドリング網羅

## 対話型レビュー手順

1. 全観点で分析し、影響の大きい課題を抽出する。
2. 課題は最大3件に絞って提示する。

```text
🔴 Critical Issue [1-3]: <題名>
Concern: <問題内容>
Impact: <影響>
Suggestion: <改善提案>
```

3. 強みを1-2件提示する。
4. GO/NO-GO を判定する。

### GO基準

- 致命的な構造不整合がない
- 要件を十分に満たしている
- 実装方針が明確
- リスクが許容範囲

### NO-GO基準

- 根本的な構造衝突がある
- 重要要件が欠落
- 実装失敗リスクが高い
- 複雑度が要求に対して過大

## 出力フォーマット

1. Design Review Summary
2. Critical Issues（最大3件）
3. Design Strengths（1-2件）
4. Final Assessment（Decision / Rationale / Next Steps）
5. Interactive Discussion（論点整理）

## 出力

- 設計レビュー結果（重大課題、強み、判定）

## レビュー指針

1. 重大課題に限定する。
2. 指摘には必ず改善提案を添える。
3. バランスを取り、強みも示す。
4. 実行可能な提案のみ出す。

## 次フェーズ

GO判定なら `spec-tasks <feature-name>` へ進む。  
自動承認で連続進行する場合は `spec-tasks <feature-name> -y` 相当を使う。

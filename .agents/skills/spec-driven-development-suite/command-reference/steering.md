# steering

## 目的

`./docs/sdd/steering/` 配下の基礎ステアリング文書を、既存内容を尊重しながら作成・更新する。

## 対象ファイル

- `./docs/sdd/steering/product.md`
- `./docs/sdd/steering/tech.md`
- `./docs/sdd/steering/structure.md`

## 入力

- 任意: 現在のプロジェクト概要、主要制約、既存ドキュメント

## 既存ファイル確認

1. `./docs/sdd/steering/` が存在しない場合は作成する。
2. 各コアファイルの存在を確認する。
   - 既存: 既存構造を維持し更新
   - 未存在: 初期生成
3. カスタムステアリング数を確認し、保全対象として扱う。

## プロジェクト分析

1. 現在のソース・設定・ドキュメントを調査する。
2. 直近変更（差分、コミット）を確認する。
3. 既存READMEや設定ファイルを文脈として読み込む。

## スマート更新戦略

### 新規ファイルの場合

- 初期版として網羅的な内容を生成する。

### 既存ファイルの場合

1. ユーザー追記・カスタム節を保持する。
2. 事実情報（依存、構成、コマンド）を更新する。
3. 新規能力がある場合のみ節を追加する。
4. 非推奨情報は削除ではなく明示的に非推奨化する。
5. 既存の書式・トーンを維持する。

## 包含モード（コアファイル）

コア3ファイルは原則 Always Included として扱う。

1. `product.md`: すべての意思決定で参照
2. `tech.md`: 技術制約の基準として参照
3. `structure.md`: 構造・命名・配置の基準として参照

## 実行手順

### 1. `product.md`

新規時:

- Product Overview
- Core Features
- Target Use Case
- Key Value Proposition

既存時:

- 新機能、廃止機能、対象ユーザー、価値提案の変更のみ更新する。

### 2. `tech.md`

新規時:

- Architecture
- Frontend（該当時）
- Backend（該当時）
- Development Environment
- Common Commands
- Environment Variables
- Port Configuration

既存時:

- 依存追加/削除、主要バージョン変更
- 開発ツール変更
- 環境変数変更
- ポート変更

### 3. `structure.md`

新規時:

- ルート構成
- 主要サブディレクトリ
- コード組織パターン
- 命名規約
- インポート規約
- 主要アーキテクチャ原則

既存時:

- 構成変更、命名変更、原則変更、再編成を反映する。

### 4. カスタムステアリングの扱い

1. 既存カスタム文書は原則保持する。
2. 既に無効化された機能への参照があれば注記する。
3. 新しい専門領域が生じたら追加候補を提案する。

## 出力

- `./docs/sdd/steering/product.md`
- `./docs/sdd/steering/tech.md`
- `./docs/sdd/steering/structure.md`

## 実行規約

1. 可能な限り事実ベースで記述し、推測で埋めない。
2. 各文書は具体例を含め、抽象論で終わらせない。
3. 追記型更新を基本とする。

## セキュリティ指針

1. 秘密情報（鍵、パスワード、接続情報、個人情報）を記載しない。
2. 共有文書であることを前提に公開可能性を確認する。
3. 版管理前に内容レビューする。

## 保全指針

1. ユーザー追加セクションは維持する。
2. ユーザー追加例は維持する。
3. インラインコメントや注記を維持する。
4. 既存のMarkdownスタイルを尊重する。

## 更新哲学

1. 置換より追記を優先する。
2. 非推奨は `[DEPRECATED]` または打消しで示す。
3. 重要変更には日時と理由を明記する。

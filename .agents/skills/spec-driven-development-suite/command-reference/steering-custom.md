# steering-custom

## 目的

領域特化のガイドライン文書を `./docs/sdd/steering/` に追加し、適用意図と優先順位を明確化する。

## 入力

- 必須: 対象トピック（例: API, testing, security, performance）
- 任意: 適用範囲、強制度（必須/推奨）、例外条件

## 前提確認

1. 既存のコアステアリングとの重複を確認する。
2. 文書管理用のInclusion Mode（Always/Conditional/Manual）を決める。

## 現状分析

1. 既存ステアリング一覧を確認する。
2. カスタム文書数を把握する。
3. プロジェクト内の専門領域（api/auth/security/test等）を抽出する。
4. 設定ファイル群から運用トピック候補を抽出する。

## 代表的なカスタム種別

1. API標準（`api-standards.md`）
   - REST/GraphQL規約
   - エラーハンドリング
   - 認証/認可
   - バージョニング
2. テスト方針（`testing.md`）
   - 配置/命名
   - モック戦略
   - カバレッジ基準
   - E2E/Unit/Integrationの使い分け
3. コードスタイル（`code-style.md`）
   - 言語別規約
   - 整形ルール
   - コメント基準
   - 命名規約
4. セキュリティ方針（`security.md`）
   - 入力検証
   - 認証方式
   - シークレット管理
   - セキュリティレビュー項目
5. DB規約（`database.md`）
   - スキーマ設計
   - マイグレーション
   - クエリ最適化
   - バックアップ/復旧
6. パフォーマンス規約（`performance.md`）
   - レイテンシ/メモリ基準
   - 最適化

7. デプロイ運用（`deployment.md`）
   - CI/CD段階
   - 環境設定
   - リリース/ロールバック
   - ヘルスチェック

## Inclusion Mode記法（文書管理用）

このスキルでは `spec-*` / `validate-*` / `spec-impl` 実行時に  
`./docs/sdd/steering/*.md` を全件読み込む。  
Inclusion Mode は「文書の意図と適用対象を人間が把握しやすくするためのタグ」として扱う。

### 1. Always Included（限定的に使用）

- 用途: 全コードへ共通適用される規約
- 影響: 本スキルは全件読み込みのため、文書数・文量が増えるほど文脈サイズが増える
- 例: 厳格なセキュリティ標準

### 2. Conditional Inclusion（推奨）

- 用途: 特定ディレクトリ/ファイル種別向け規約
- 例: `"*.test.ts"`, `"src/api/**/*"`, `"**/auth/*"`, `"*.config.*"`
- 利点: 適用対象が明確になり、レビュー判断が速くなる

### 3. Manual Inclusion

- 用途: 低頻度の専門運用
- 使い方: レビューや設計議論で `@filename.md` を使って焦点指定する
- 利点: 低頻度ルールの存在を明示しつつ、運用意図を共有できる

## 文書構造ガイド

1. 明確なタイトルと目的
2. 具体的な規約と判断根拠
3. 正例（必要なら反例）
4. 他ステアリングとの関係

## 実行手順

1. `./docs/sdd/steering/<topic>.md` を作成する。
2. 適用対象、必須ルール、非推奨パターン、例外条件を明記する。
3. 既存文書との優先順位を文書末尾に記載する。
4. 先頭コメントで Inclusion Modeタグを明示する。

```markdown
<!-- Inclusion Mode: Always | Conditional: "pattern" | Manual -->
```

## 出力

- `./docs/sdd/steering/<topic>.md`

## セキュリティ・品質基準

1. 機密情報は記載しない。
2. 内部限定情報（プライベートURL等）は極力避ける。
3. 1ファイル1責務を守る。
4. 2-3分で読めるサイズを目安にする。
5. 他文書との重複を避け、固有価値を持たせる。

# Agent Handoff Plan

## Overview

設計・方針が固まった作業を、別の実装エージェントが単独で完遂できる形に落とし込むワークフロー。
成果物は 2 つ: (1) 計画書（Markdown）、(2) そのままコピーして実装エージェントに渡せるハンドオフプロンプト。

このスキルの品質基準は「実装エージェントが計画書だけを読んで、途中でユーザーに質問せずに完了できるか」。
判断の余地が残っている箇所は、計画書に書く前にユーザーに確認して確定させる。

実装完了後の検証・差し戻しは `agent-review-request` スキルの領域なので、
計画書とプロンプトを渡した時点でこのスキルの仕事は終わり。

## Workflow

### 1. 前提確認

- 設計・採用案が確定しているかを確認する。UI 案の選択が済んでいない、方式の比較検討が
  終わっていない、といった状態なら、先にそちらを完了させる（必要なら ui-mock-proposals や
  通常の設計議論に戻る）。
- 現状のコードを実際に読み、計画書に書く内容が現実と一致していることを確認する。
  推測で書いた計画書は実装エージェント側で矛盾として顕在化し、差し戻しコストが倍になる。
- 曖昧な点・設計判断が必要な点が残っていたら、計画書を書き始める前にユーザーに確認する。
  「実装エージェントに判断させる」選択肢は原則取らない（判断が割れると検証で手戻りになるため）。

### 2. 計画書の作成

**置き場所と命名**: プロジェクトに計画書の置き場所の規約があればそれに従う
（例: `docs/plans/`、`docs/superpowers/plans/`）。なければ
`~/dotfiles/tmp/claude/YYYY-MM-DD-<project>-plan.md` に置く。
同日に複数作る場合は `-plan-02` のように連番を付ける。

**構成**: 次のセクションを必ず含める。

```markdown
# <タイトル>

## 背景と目的
（なぜこの変更をするか。現状の問題点）

## 確定済みの設計判断
（採用案とその理由。検討して不採用にした案があれば理由付きで明記。
 実装エージェントが「こっちの方が良いのでは」と迷わないようにする）

## 実装フェーズ
（Phase 1, 2, ... に分解。各フェーズに:
 - 対象ファイルと変更内容（ファイルパスは実在確認済みのものを書く）
 - そのフェーズの完了条件）

## 参照物
（HTML モック・スクリーンショット・関連ドキュメントのローカルパス。
 モックが仕様源になる場合は「このモックが唯一の仕様源」と明記し、
 モック内のどの要素がどのコード・トークンに対応するかを書く）

## DoD（Definition of Done）
- [ ] 機能完了条件（測定可能な形で列挙）
- [ ] テスト完了条件（実行するコマンドと期待結果）
- [ ] 運用反映条件（lint / typecheck / build の通過、必要ならドキュメント更新）

## やらないこと
（スコープ外を明示。実装エージェントの善意の拡大解釈を防ぐ）
```

- DoD のない計画書は未完成として扱う。3 分類（機能・テスト・運用反映）を必ずチェックリストで書く。
- HTML モックなどの参照物は必ずローカルに保存してからパスを計画書に書く。
  会話内にしか存在しない成果物は実装エージェントから見えない。

### 3. 計画書のセルフレビュー

書き終えたら渡す前に一度見直す。観点:

- 実装エージェントがユーザーに質問しないと進められない箇所が残っていないか
- ファイルパス・シンボル名・色トークンなどの具体値が現実のコードと一致しているか
- フェーズ間の依存関係が明示されているか（並行可能なら並行可能と書く）

### 4. ハンドオフプロンプトの生成

計画書とは別に、実装エージェントの入力欄にそのまま貼れるプロンプトを
コードブロックで出力する。テンプレート:

```
対象リポジトリ: <絶対パス>

<計画書の絶対パス> を読んで、この計画書に従って実装してください。
AGENTS.md / CLAUDE.md のルールに従ってください。

- 計画書の「確定済みの設計判断」は変更しない。実装中に矛盾や不明点を見つけた場合は、
  自分で判断せず、その箇所を報告して停止する
- 「やらないこと」に書かれた範囲には手を出さない
- 完了したら DoD チェックリストの各項目について、実行したコマンドと結果を添えて報告する
```

- 実装エージェント側に effort / reasoning レベルの指定がある場合（ユーザーが high / xhigh
  などを指定したとき）はプロンプト冒頭に添える。指定がなければ勝手に付けない。
- 対象が worktree の場合はそのパスを使う（メインの checkout と取り違えない）。

### 5. 引き継ぎ後

- 計画書のパスとハンドオフプロンプトを提示して完了。実装の進捗監視はしない。
- ユーザーが「実装が終わったから確認して」と戻ってきたら `agent-review-request` スキルで
  受け入れ検証を行う（計画書パスを検証の入力として渡す）。

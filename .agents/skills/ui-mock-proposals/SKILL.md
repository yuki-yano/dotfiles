# UI Mock Proposals

## Overview

UI の見た目に関する改善案を、比較可能な単一の HTML モックとして複数案提示し、
ユーザーの選択と反復を経て実装（または実装引き継ぎ）につなげるワークフロー。

典型的な依頼は「現状のスクリーンショット + 不満点の箇条書き + 複数案を HTML で」という形で来る。
流れ: 実装調査 → 複数案を 1 つの HTML に作成 → プレビュー提示 → 選択 → 微調整の反復 → 次工程へ。

## Workflow

### 1. 事前調査（モックを書く前に必ず行う）

- 対象 UI の実装を読み、実際の色トークン（CSS 変数、テーマパレット）、フォント、アイコン、余白の値を把握する。
  推測で配色案を出すと、後から実装を見て矛盾に気づき案を出し直す手戻りになる。
- 実装対象の技術的制約を確認する。特に TUI（tmux statusline 等）が対象の場合、
  box-shadow・グラデーション・角丸など端末で再現できない装飾は案に含めない。
  HTML モックはあくまで完成イメージであり、実装できる表現の範囲で描く。
- スクリーンショットが添付されている場合は、まず「現状」を忠実に再現したセクションを作り、比較の基準にする。

### 2. 案の設計

- 3〜5 案 + 「現状」を用意する。各案は方向性が明確に異なるようにする
  （例: ミニマル / リッチ / 情報密度重視。同じ方向の微差 3 案は比較の意味がない）。
- ゼロベースで考える。現状デザインへの局所修正だけで案を埋めない。
  ユーザーが「ゼロベースで」「フラットに評価して」と言ったときは、
  現状を前提にした評価軸そのものを捨てて作り直す。
- 推奨案は 1 つだけ明示してよいが、あくまで対等な選択肢として提示する。
  ユーザーは推奨と異なる案を選ぶことがあり、それが正しい結末になる。

### 3. HTML の作成規約

- 全案を 1 ファイルにまとめる。案ごとにファイルを分けない（並べて比較できることが目的）。
- 命名は `<対象>-proposals.html`、配色などの比較が主目的なら `<対象>-compare.html`。
- 置き場所はそのプロジェクトに適したパスを選ぶ。使い捨ての比較なら `tmp/` など
  gitignore 済みの作業ディレクトリ、採用案を記録として残す流れなら `docs/` 配下。
  プロジェクトに既存のモック置き場があればそれに従う。
- 外部リソースなしで開ける自己完結ファイルにする。Tailwind 等の CDN は使わず、
  `<style>` タグ 1 つに素の CSS を書く。
- 実アプリの色トークン・フォントを移植し、「そのまま採用できる見た目」の再現度にする。
- アプリがライト/ダーク両対応なら、モックも `prefers-color-scheme` と
  `data-theme` 上書きの両方でテーマ対応する。
- 各案には案 ID（案A/案B… または 案1/案2…）の見出し、方向性の一言説明、
  短い Pros/Cons を付け、HTML を見るだけで選択できるようにする。
- 表示するデータはダミー文字列ではなく、実アプリに出る現実的な内容を使う。

### 4. 提示と選択

- 配置した HTML を `open <path>` でブラウザ表示して提示するのを基本とする。
  Artifact ツールは基本的に使わない（ユーザーが明示的に求めた場合のみ）。
- 反復で更新するときは同じファイルを上書きし、ユーザーにはリロードで確認してもらう。
- 案の選択に AskUserQuestion は使わない。ユーザーは HTML を見て
  「Bで」「A+D で」のように自由回答で選ぶ。テキスト択一の提示は
  「HTML で比較させて」と拒否された実績がある。
  AskUserQuestion を使ってよいのは、コントラスト比や要件など実装細部の確認に限る。

### 5. 反復

- 一発では決まらない前提で臨む。「色をもう少し弱く」「案 B ベースで
  この部分のバランスを直して」のような微調整依頼には、同じファイルを更新して応える。
- 方向転換の指示が来たら、既存案に固執せず作り直す。
- 明確な選択がないまま話題が変わることもある。その場合は選択を強要しない。

### 6. 選択後の次工程

選択後の進め方は 2 パターンあり、ユーザーの指示がなければどちらに進むか確認する。

- **直接実装**: 選ばれた案をそのまま実装し、テスト・lint を通す。
- **実装引き継ぎ**: Codex 等の別エージェントに実装させるための計画書
  （`docs/plans/YYYY-MM-DD-plan-NN-<対象>.md` 等）を書いて渡す。

いずれの場合も、選ばれた案のモックが唯一の仕様源になるので、
モックと実装対象コードの対応（どのトークン・どのコンポーネントに反映するか）を明記する。
